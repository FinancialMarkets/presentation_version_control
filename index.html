What is Version Control?

Version control is a method of managing, and keeping track of, changes to code. Of course, code is simply text, so version control can be applied to writing books, etc.

An Analogy

If you are familiar with Microsoft Word, you can think of version control as a very advanced track-changes system.

Beyond Track Changes Version Control Provides:

  • All historical changes with who made them and when.
  • The ability to revert any change in the history.
  • The ability to have multiple versions and merge any set of versions.
  • Among many others.

Types of Version Control

There are various version control implementations such as:

  • Git
  • Subversion
  • CVS

However we'll only discuss Git, which is presently the most popular version control system (VCS).

Git

Git is standalone software which implements a VCS.

Git and IDEs

IDEs, such as VSCode, provide Git integration so you can perform common version control (Git) tasks from within the IDE. This means:

  • You can use the mouse.
  • You only have to know what you want to do, and the general command for it.
  • You don't have to know the exact commands.
  • You are limited to a subset of git commands, however. To do more complicated tasks will require a shell (terminal interface).

Workflow and Important Commands

In the next few slides will cover some important commands to get you started using git for version control. What follows provides the git commands you would execute in a shell. If you are using an IDE (which is recommended) you only need to look for the button for the command.

Starting a Git Repo

To start version control in a directory you 'initialize' the git repository with:


$ git init

This will create a hidden directory named .git

Tracking Files

Initially, no files are under version control---Git is not watching them for changes. To track them we need to add them with (say we want to track a file named example.py):


$ git add example.py

Committing

We then commit the changes, which is like setting a market in the life of the file.


$ git commit -m "initial commit"

The "initial commit" is the commit message. You can put in whatever description of the commit you think is most appropriate.

Git Status

The git status command will tell you the present state of the repository, which includes information such as any files with unstaged or staged changes. It will also indicate the latest commit (known as Head).


	      $ git status

git diff

Say example.py has unstaged changes. We can take a look at the changes with:


	      $ git diff example.py

Staging Changes

If we want to commit the changes we first stage the file with git add and then commit with git commit. Note there is no git stage command. We use git add for both initially tracking a file and later staging changes.


    $ git add example.py
    $ git commit -m "a description of the changes"

In this way you will have a history of every change made to a file or set of files. You can also, at any time, see who wrote each line in a file by using:


	      $ git blame example.py

There is also a great deal of other things you can do, like undoing changes and branching, however the above is enough to get you started, with the exception of learning how to host your code online. We'll cover this next.
